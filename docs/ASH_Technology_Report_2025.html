Step-by-Step Implementation Guide

ARIA Setup

Prerequisites

Install Python 3.11+, Node.js 18+, and Ollama. Install wkhtmltopdf for PDF export capability.

Clone & Configure

Clone the repository. Copy .env.example to .env. Generate a JWT secret with python -c "import secrets; print(secrets.token_hex(32))". Set ADMIN_PASSWORD to a strong password (12+ characters).

Ollama Model Setup

Start Ollama with ollama serve. Pull the chat model: ollama pull llama3.2:1b. Pull the embedding model: ollama pull nomic-embed-text. Verify with ollama list.

Backend Setup

Create virtual environment: python -m venv venv. Activate it. Install dependencies: pip install -r requirements.txt. Place policy PDF documents in the configured POLICY_DOCS_DIR directory.

RAG Ingestion

On first startup, ARIA automatically ingests all PDF documents from POLICY_DOCS_DIR into ChromaDB. The system chunks documents (1000 chars, 200 overlap) and generates embeddings via nomic-embed-text. This process runs once; subsequent starts load from persistent storage.

Start Backend

Run uvicorn main:app --host 0.0.0.0 --port 8000 --reload. The startup sequence validates secrets, initializes the SQLite database, creates the default admin user, and starts the APScheduler for 6-hour monitoring cycles. Verify at http://localhost:8000/docs.

Frontend Setup

Navigate to frontend/. Run npm install. Start development server: npm run dev. The Next.js proxy automatically rewrites /api/* requests to the backend at port 8000. Access at http://localhost:3000.

Verify

Login with configured admin credentials. Navigate to Expert Chat and ask a policy question. Verify RAG retrieval (responses should cite specific policy documents). Check Settings page for monitoring agent status. Generate a test policy and run compliance review.

CyberDash Setup

Prerequisites

Install Docker and Docker Compose. Install Ollama. Ensure ports 80, 443, 5000, 5432, 6379, 11434 are available.

Clone & Configure

Clone the repository. Copy .env.example to .env. Set SECRET_KEY, ADMIN_PASSWORD_HASH (bcrypt hash), DATABASE_URL, and REDIS_URL. Optionally configure TELEGRAM_BOT_TOKEN for ops workflow.

Pull LLM Model

Pull the AI model: ollama pull llama3.2:1b. The system auto-detects available models and falls back to the smallest variant if the configured model isn't found.

Docker Compose Launch

Run docker compose up -d --build. This launches all 8 services: backend, frontend, PostgreSQL, Redis, Ollama, Nginx, Certbot, and n8n. Health checks ensure services start in correct order (database before backend, Ollama before AI jobs).

Initial Data Ingestion

The scheduler automatically triggers the first RSS feed fetch. Alternatively, use the Admin Panel to manually trigger POST /admin/fetch. The system ingests, deduplicates, and filters articles for cybersecurity relevance.

AI Draft & Publish

Access the Admin Panel. Select news items from the queue. Click "AI Draft" to generate CISO-ready summaries. Review and edit the AI output. Click "Finalize" for fact-checking. Approve and publish to the executive dashboard.

Telegram Bot (Optional)

Create a Telegram bot via @BotFather. Set TELEGRAM_BOT_TOKEN in .env. Send any message to the bot — the system auto-discovers and registers your chat ID. Daily briefings will be sent at 06:00 UAE time.

Verify

Access the dashboard at http://localhost. Check the GeoPulse Atlas for threat visualization. Verify KPI cards show live metrics. Test the intelligence chat. Run GET /admin/system/health to confirm all components are operational.

Future Improvements

Strategic roadmap for enhancing both platforms based on technical audit findings and operational requirements.

ARIA Improvements
P0 — Critical

Migrate to langchain_ollama Imports

Current code uses deprecated langchain_community.ChatOllama and OllamaEmbeddings. Migrate to langchain_ollama package for continued compatibility and performance improvements.

P0 — Critical

Persistent Audit Log Database

Currently audit events are only logged to stdout. Implement a dedicated audit_logs SQLite table capturing all authentication events, policy operations, and monitoring cycles for compliance evidence.

P1 — High

Role-Based Access Control (RBAC)

Extend single-admin model to support multiple roles: CISO (full access), Policy Author (generate/review), Auditor (read-only compliance views), Viewer (dashboard only).

P1 — High

PostgreSQL Migration

Migrate from SQLite to PostgreSQL for multi-instance deployment, concurrent access, and enterprise backup/restore capabilities. Required for horizontal scaling.

P1 — High

Distributed Rate Limiting

Replace in-memory rate limit store with Redis-backed implementation. Current approach loses state on restart and doesn't work across multiple API instances.

P2 — Medium

Docker Compose Deployment

Create containerized deployment matching CyberDash's Docker Compose architecture. Include Ollama, backend, frontend, and Nginx services with health checks and persistent volumes.

P2 — Medium

Policy Version Control

Implement version history for generated policies. Track changes, enable diff comparison, and maintain approval chain records. Essential for ISO 27001 document control requirements.

P2 — Medium

Larger LLM Model Support

Add configuration profiles for larger models (llama3:7b, codellama) with automatic context window management. Larger models significantly improve policy generation quality and compliance review depth.

P3 — Future

Multi-Tenant Architecture

Enable ARIA to serve multiple business units within Ali & Sons Group, each with isolated policy libraries, ChromaDB collections, and compliance requirements.

P3 — Future

Policy Gap Analysis Dashboard

Visual dashboard showing compliance coverage heatmap across all 7 frameworks. Identify which policies need updating, which framework controls lack coverage, and track remediation progress.

CyberDash Improvements
P0 — Critical

API Key Management

Implement API key authentication for programmatic access (n8n workflows, SIEM integrations). Current basic auth is not suitable for automated service-to-service communication.

P0 — Critical

Input Sanitization Hardening

Add comprehensive XSS prevention on all user-submitted content (news items, highlights, posture notes). Implement Content Security Policy (CSP) headers on Nginx.

P1 — High

P1 — High

RAG-Powered Intelligence Chat

P2 — Medium

P2 — Medium

Historical Trend Analytics

Add time-series visualization of threat categories, severity distribution, and source coverage over 30/60/90-day windows. Enable trend-based risk assessment for board reporting.

P2 — Medium

Multi-Language Support

Add Arabic language support for executive briefings and dashboard labels. Essential for UAE government stakeholder communications and ADDA compliance.

P3 — Future

Threat Intelligence Sharing (STIX/TAXII)

Implement STIX 2.1 format export for threat indicators. Enable TAXII server for automated intelligence sharing with UAE CERT and sector ISACs.

Cross-Platform Integration Roadmap

P1 — High

ARIA ↔ CyberDash Bidirectional Integration

Connect both platforms to create a closed-loop security governance system:

CyberDash → ARIA: When CyberDash publishes a critical threat briefing, automatically trigger ARIA's Policy Writer to draft or update the relevant policy (e.g., new ransomware variant triggers Incident Response Policy review).

ARIA → CyberDash: When ARIA's Autonomous Monitor detects a regulatory update (e.g., new NESA control), push a notification to CyberDash's leadership highlights board for C-suite visibility.

Shared Vector Store: Unify ChromaDB collections so CyberDash intelligence enriches ARIA's policy context, and ARIA's policy knowledge grounds CyberDash's remediation recommendations.
